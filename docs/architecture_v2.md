# GapFinder Architecture v2

> 2026-04-13 CEO 인풋 기반 구조 재설계

---

## 1. 설계 원칙

- **Platform level**: 다양한 주제·독자·투자 철학을 커버하는 범용 엔진
- **Article level**: 반드시 1 타깃 독자 + 1 목적 + 1 thesis로 강제
- CEO는 "무엇을 말할지" 결정, AI는 "어떻게 만들지" 실행
- 초기에는 CEO 인풋 의존 → 중기 이후 AI가 자동 제안까지 고도화

---

## 2. 역할 분담

| 역할 | 담당 | 범위 |
|------|------|------|
| CEO | 주제 방향, 대략적 결론, 투자 철학, 글별 인풋, 최종 승인 | 판단·기획 |
| AI (시스템) | 앵글 제안, 데이터 수집, 논리 구조화, 초안 작성, 제목/썸네일, 이미지 | 실행·자동화 |
| CTO | 시스템 구축, 모듈 개발, DB 관리, 파이프라인 운영 | 개발·인프라 |

---

## 3. 모듈 구성 (10개)

### Core Pipeline (순차 실행)

| # | 모듈 | 역할 | 입력 → 출력 |
|---|------|------|-------------|
| 0 | **Topic Scout** | 시장 데이터·뉴스·커뮤니티 스캔 → 글감 후보 제안 (매일/수시) | 3층 DB → 글감 후보 리스트 |
| 1 | **Brief Manager** | CEO 인풋 수집·저장. 후기에는 AI가 Brief 초안 자동 제안 | CEO 지시 → 정형화된 Brief |
| 2 | **Research Engine** | Brief 기반 데이터 수집. 3층 DB + 웹 서치 | Brief → 관련 데이터·팩트 패키지 |
| 3 | **Angle Generator** | 추가 앵글·반론·보완 논점 제안 | Brief + 데이터 → 앵글 후보 리스트 |
| 4 | **Structure Builder** | 논리 뼈대 설계 (Thesis → 근거 → 반론 처리 → 결론) | 확정 앵글 + 데이터 → 아웃라인 |
| 5 | **Writer** | 아웃라인 기반 초안 작성 (톤·스타일·분량을 Brief에 맞춤) | 아웃라인 → 초안 |
| 6 | **Packager** | 제목, 부제, 썸네일 문구, 이미지 생성 | 초안 → 발행용 패키지 |

### Quality & Control (파이프라인 전반에 개입)

| # | 모듈 | 역할 | 적용 시점 |
|---|------|------|-----------|
| 7 | **Rule Engine** | Hard rule 체크 — 통과 못하면 발행 불가 | 구조화 후 + 초안 후 |
|   | | - Thesis가 하나인가 (단일 thesis 강제) | |
|   | | - Transmission mechanism 완성됐는가 (A→B→C→결론) | |
|   | | - 데이터 근거가 있는 주장인가 | |
|   | | - 타깃 독자 수준과 톤이 일치하는가 | |
| 8 | **Scoring Engine** | Soft score 평가 — 발행 판단 보조 | 초안 완성 후 |
|   | | - 논리 완성도 (0-100) | |
|   | | - 차별화 정도: 컨센서스와 얼마나 다른가 | |
|   | | - 읽힘성: 독자가 끝까지 읽을 확률 | |
|   | | - 데이터 신선도: 인용 데이터가 최신인가 | |

### Learning & Knowledge (지속 축적)

| # | 모듈 | 역할 | 축적 방식 |
|---|------|------|-----------|
| 9 | **Reference DB** | 좋은 글 레퍼런스, 투자 철학 프레임, 스타일 패턴 저장 | CEO가 좋은 글 등록 + 태깅 |
| 10 | **Feedback Loop** | CEO 피드백 수집 → Rule/Scoring 보정 + Brief 자동제안 고도화 | 매 글 발행 후 CEO 리뷰 |

## 4. CEO Input Brief 포맷

### 필수 입력 (Required)

| 필드 | 설명 | 예시 |
|------|------|------|
| **Thesis** | 핵심 주장 한 문장 | "오라클 부채는 RPO 기반이라 오히려 저평가" |
| **Why Now** | 왜 지금 이 글이어야 하는가 | "실적 발표 직후 부채 우려로 -8% 급락" |
| **타깃 독자** | 누구에게 쓰는 글인가 | 미국 개별주 투자하는 개인투자자 |
| **콘텐츠 타입** | 어떤 형태의 글인가 | 심층 분석 / 숏폼 / 뉴스 코멘터리 / 교육 |

### 선택 입력 (Optional — 안 주면 AI가 제안)

| 필드 | 설명 | 예시 |
|------|------|------|
| **투자 철학 프레임** | 어떤 관점으로 볼 것인가 | 역발상 / 가치투자 / 모멘텀 / 퀄리티 |
| **톤** | 글의 성격 | 공격적 주장 / 균형 분석 / 교육적 |
| **반드시 포함할 포인트** | 빠지면 안 되는 내용 | "RPO $130B 언급 필수" |
| **반드시 피할 포인트** | 쓰면 안 되는 내용 | "AI 버블 프레이밍 금지" |
| **참고 자료** | CEO가 직접 던지는 소스 | 링크, 기사, 트윗 등 |

### 플로우

```
Topic Scout가 글감 제안 (매일/수시)
       ↓
CEO가 주제 선택 or 직접 주제 입력
       ↓
CEO가 필수 4개 입력
       ↓
AI가 선택 필드 초안 제안
       ↓
CEO 승인/수정
       ↓
파이프라인 실행 (Research → Angle → Structure → Write → Package)
       ↓
Rule Engine + Scoring Engine 체크
       ↓
CEO 최종 승인 → 발행
       ↓
Feedback Loop에 피드백 저장
```

| 단계 | CEO | AI |
|------|-----|-----|
| **초기** | Brief 전부 작성 | 실행만 |

| **후기** | 승인만 | 시장 스캔 → 주제+Brief 자체 제안 |
