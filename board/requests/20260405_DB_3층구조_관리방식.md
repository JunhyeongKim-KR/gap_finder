# DB 3층 구조 관리 방식

## CEO 핵심 지시
> "DB는 기사 저장소가 아니라, 원문 저장층 + 숫자/지표를 담는 정형층 + 투자 아이디어/해석을 담는 노트층으로 나눠서 관리해야 한다."

## 3층 구조

### 1층: 원문 저장소
- 기사/공시/IR/리포트/정책 원문을 **손대지 않고 그대로 저장**
- 포맷: raw_text, html, pdf path, markdown
- 목적: 근거 확인, 재파싱, 사람 검토

### 2층: 정형/반정형 분석 DB
- 숫자와 메타데이터를 **칼럼형 테이블**로 저장
- Agent가 글 쓸 때 바로 꺼내 쓰는 재료
- 비교, 필터링, 시계열 분석 가능

### 3층: 지식 메모/피처 레이어
- "왜 중요한지"를 **JSON/노트**로 저장
- 철학 프레임 연결, 컨센서스, 반박 포인트, 체크포인트
- thesis card 형태

## 포맷 설계 원칙
- **숫자는 칼럼으로** (ROIC, CAPEX, 금리 등)
- **해석은 JSON/노트로** (왜 중요한지, 리스크 등)
- **원문은 별도 보관** (재파싱 가능하게)
- **태그는 필수** (company, sector, topic, event_type, frame)

## CEO 추천 8개 테이블
1. `documents` — 원문 저장
2. `entities_companies` — 회사 마스터
3. `company_fundamentals` — 재무 시계열 (ROIC/WACC/재투자율 포함)
4. `macro_series` — 거시 지표
5. `industry_metrics` — 산업 구조 (CAPEX/D&A/IPO/가동률)
6. `events` — 이벤트 마스터 (실적/유증/M&A/정책/관세/분쟁)
7. `analytical_notes` — 해석 메모/thesis card
8. `style_benchmarks` — 스타일 학습용

## 글 생성용 핵심 포맷 2가지
1. **Fact table** — 비교/조회 가능한 숫자 (8분기 ROIC, CAPEX 추이 등)
2. **Thesis card** — 투자 아이디어 카드 (consensus, counter_view, mechanism, checkpoints)

## CEO 제안 구현 순서
- 초기: documents, entities_companies, events, macro_series, analytical_notes
- 2차: company_fundamentals, industry_metrics, style_benchmarks
- 3차: 벡터검색/RAG, 스타일 최적화, 자동 thesis card 생성
