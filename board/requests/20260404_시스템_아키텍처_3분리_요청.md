# CEO 요청: 시스템 아키텍처 3분리 설계

> 수신일: 2026-04-04
> 발신: CEO → CTO
> 상태: 검토 대기

---

## 요약

CEO가 GapFinder 시스템을 **구현 관점에서 3개 레이어로 분리**하여 재정리함.
핵심 메시지: "Agent는 생각하는 법을 알고, DB는 사실을 제공하고, 시스템은 그 둘을 연결해서 역발상 분석형 글을 안정적으로 쓰게 만드는 구조"

---

## 1. Agent 프롬프트 레이어 (고정 — 잘 안 바뀌는 것)

AI가 항상 깔고 가야 하는 사고방식/투자 철학/글쓰기 원칙.
데이터가 바뀌어도 **해석 관점은 고정**.

### 1-1. 블로그 정체성 & 글의 기본 문법

단순 뉴스 요약이 아닌, 역발상 분석형 글.

**고정 문법:**
```
[시장 컨센서스] → [해석의 맹점] → [실제 메커니즘] → [진짜 수혜/피해] → [체크포인트]
```

한 줄 요약: "시장은 A라고 보지만, 실제 병목과 구조는 B이므로, 진짜 수혜는 C다."

### 1-2. 투자 철학 프레임 (22개)

#### A. 산업/거시 해석 (8개)

| # | 프레임 | 핵심 |
|---|--------|------|
| 1 | 자본 사이클형 | 수요보다 공급을 먼저 봄. CAPEX/D&A, 신규진입, IPO 추적 |
| 2 | 경쟁우위/진입장벽형 | 초과수익이 왜 유지되는지. 브랜드/전환비용/네트워크 효과 |
| 3 | 제도·인프라 설계형 | 기업보다 거버넌스 구조를 먼저 봄. 정부주도 vs 민간자율 |
| 4 | 생태계 지속가능성형 | 확산 속도보다 참여자 생존 가능성. 이탈률/과금/규제 부담 |
| 5 | 데이터 거버넌스형 | 데이터 소유권/이동/허브/lock-in 구조 |
| 6 | 자원·에너지 패권형 | 기술 뒤에 자원/전력 제약. 누가 싸게 확보하는가 |
| 7 | 지정학 블록 재편형 | 에너지/안보/기술 중심 블록 재편. 한국의 역할 |
| 8 | 병목 인프라형 | 표면 테마보다 병목을 쥔 업체가 진짜 수혜 |

#### B. 기업/종목 해석 (8개)

| # | 프레임 | 핵심 |
|---|--------|------|
| 9 | 기업가치 평가형 | 주가가 아니라 사업가치 대비 가격 |
| 10 | 자본배분형 | 실적보다 돈 쓰는 방식 (CAPEX/M&A/배당/자사주/현금) |
| 11 | 진짜 현금흐름형 | PER보다 FCF/owner earnings. 유지vs성장 CAPEX 구분 |
| 12 | 해자 지속성형 | 해자 유무보다 해자 침식 여부 |
| 13 | 퀄리티 복리형 | 높은 ROIC의 지속가능성 + 재투자 활주로 |
| 14 | 경영자 DNA형 | 오너십/인센티브/보상구조/TSR연동 |
| 15 | 데이터 인프라 수혜형 | AI 자동화 → 데이터 인프라 수요 증가 |
| 16 | 성장형 부채 해석형 | 부채 = 생산적 투자? RPO/백로그/수주 확인 |

#### C. 가격/기대 철학 (3개)

| # | 프레임 | 핵심 |
|---|--------|------|
| 17 | 기대치 역산형 | 현재 주가가 반영한 기대치부터 역산 |
| 18 | 내러티브/역발상형 | 시장의 과도한 해석 → 반대 투자 아이디어 |
| 19 | 하방 방어형 | 하락장 방어가 장기 복리의 핵심 |

#### D. 사고 체계 (3개)

| # | 프레임 | 핵심 |
|---|--------|------|
| 20 | 프로세스·확률형 | 결과보다 판단의 질. 기저율 → 확률 업데이트 |
| 21 | 리스크 관리형 | 리스크 = 변동성이 아니라 영구적 자본 손실 가능성 |
| 22 | 멘탈모델 통합형 | 심리/통계/인센티브/공급/복리를 함께 고려 |

### 1-3. 글쓰기 원칙 (10개)

1. 단순 뉴스 요약 금지
2. 항상 시장 컨센서스를 먼저 제시
3. 그 해석의 맹점을 반드시 짚기
4. 실제 메커니즘을 구조적으로 설명
5. 진짜 수혜/피해를 구분
6. 앞으로 확인할 체크포인트 제시
7. 실패 시나리오를 반드시 포함
8. 사실/해석/가설적 추론을 분리
9. 과도한 확정 표현 금지
10. 공개형·단방향·비개별적 글쓰기 유지

## 2. 데이터/DB/크롤링 레이어 (동적 — 수집·저장 대상)

Agent가 해석할 재료. 철학은 프롬프트에, 사실은 DB에.

> CEO 강조: "이 데이터 항목들은 단순 메모가 아니라, 실제 저장 스키마 설계 대상"

### 2-1. 데이터 범주 (A~J, 10개 카테고리)

| 범주 | 내용 | 대표 항목 |
|------|------|-----------|
| A. 거시/국제정세 | 금리, 물가, 환율, 유가, 관세, 외교 | CPI/PPI, 신용스프레드, GDP, ISM |
| B. 산업 구조/공급 | CAPEX, D&A, IPO, M&A, 가동률 | CAPEX/D&A 비율, 신규진입자, 재고 |
| C. 정책/제도/거버넌스 | 법적 구조, 허가제, API 표준 | 사업자 이탈률, 과금체계, 제재사례 |
| D. 자원/에너지/광물 | 우라늄, 구리, 리튬, 원전, SMR | 광산 리드타임, 송전망 투자 |
| E. 기업 재무/가치 | 매출~FCF, 밸류에이션 | ROIC, WACC, 유지/성장 CAPEX |
| F. 경영진/자본배분 | CEO 지분, 내부자 매매, 보상 | TSR 연동, M&A 이력, 주주서한 |
| G. 기대치/컨센서스 | 애널리스트 전망, 가이던스 | 목표주가 분포, 서프라이즈/쇼크 |
| H. 이벤트 대응 | 이벤트 → 시장 반응 → 후속 | 기저율, 경쟁사 행동, 컨센서스 조정 |
| I. 데이터/AI/클라우드 | I-1 인프라형 + I-2 선행수요형 | 쿼리량, RPO, 백로그, 고객 락인 |
| J. 스타일/콘텐츠 개선 | 제목, 길이, 반응, 조회수 | 도입방식, 논리전개, 스타일-성과 관계 |

### 2-2. 크롤링 2종 분리

| 종류 | 목적 | 우선순위 |
|------|------|----------|
| 글 생성용 배경지식 | 기사/공시/실적/정책/지표 수집 | **1순위 (필수)** |
| 모델 개선용 스타일 | 블로그 조회수/스타일 학습 | 2순위 (가능하면) |

## 3. CTO에게 질문 & 요청

### 3-1. 질문 4건

| # | 질문 | CEO 의견 |
|---|------|----------|
| Q1 | Agent 프롬프트와 DB를 분리하는 구조가 맞는지? | "무조건 분리해야 한다" |
| Q2 | 배경지식 크롤링과 모델 개선용 크롤링을 분리할 수 있는지? | "목적이 다르니 분리" |
| Q3 | 블로그 링크+조회수/스타일 학습 구조가 가능한지? | "가능하면 하고 싶음" |
| Q4 | 정리한 데이터 항목들이 실제 스키마 설계 대상인지? | "단순 참고가 아님" |

### 3-2. 답변에 따른 요청 (Case 분기)

**Case A (스타일 학습 가능 시):**
1. 배경지식 크롤링 파이프라인 구축
2. 스타일/벤치마크 크롤링 파이프라인 별도 구축
3. 글 스타일+조회수 기반 성과학습 구조 설계
4. Agent 글쓰기 튜닝에 반영

**Case B (배경지식만 가능 시):**
1. 글 생성용 배경지식 크롤링부터 구축
2. DB/스키마 안정화
3. Agent에 철학/글쓰기 원칙 탑재
4. 스타일 최적화는 보류

### 3-3. CEO 제안 우선순위

| 단계 | 내용 |
|------|------|
| 1단계 | Agent 프롬프트 설계 (철학+글쓰기 원칙) |
| 2단계 | 배경지식용 크롤링/DB 구축 |
| 3단계 | Agent + DB 연결 → 실제 글 생성 |
| 4단계 | 스타일/조회수 기반 개선 (가능 여부 검토 후) |

---

## CEO 최종 요약

```
철학(프롬프트) = 해석 엔진
데이터 스키마(DB) = 철학이 작동할 수 있도록 재료를 저장하는 구조
시스템 = 둘을 연결해서 역발상 분석형 글을 안정적으로 생산
```
