# 블로그 운영 시스템 보고서 — 최신 방향성

*— 어떻게 만들 것인가 / 누가 무엇을 결정할 것인가 / 무엇을 어떤 순서로 만들 것인가 —*

> 본 문서는 CEO가 제시한 최신 시스템 방향성. 기존 board/requests/ 누적 요청 위에 통합 갱신을 의도함.
> CTO/Claude는 본 문서 기반으로 의사결정·우선순위 재조정 후 작업 시작.

---

## 📌 5분 요약

### 만들려는 것

**시장 통념을 한 단계 더 의심하는 분석 블로그.** 운영자(CEO)가 글 1편당 5~10분만 투입해도 발행되는 자동화 시스템. 주 2~3편 발행.

### 시스템 흐름 (5단계)

```
[1단계 — 글감 잡기]    어떤 종목·이슈로 글 쓸지 후보 3개 생성
                       ↓ CEO가 1개 픽
[2단계 — 분석 관점]    시장이 보는 view를 반박하는 thesis 후보 3개 생성  ← 가장 핵심
                       ↓ CEO가 1개 픽
[3단계 — 근거 수집]    공시·재무·뉴스 등 데이터 자동 수집
                       ↓ 자동
[4단계 — 글 작성]      본문 초안 작성
                       ↓ CEO 1차 검수
[5단계 — 다듬기]       팩트체크 + 제목·후킹 강도 변형 3개
                       ↓ CEO가 1개 픽
                       ↓
                       발행
```

### CEO는 콘텐츠만, CTO는 시스템만

이 원칙이 핵심. CEO는 "어떤 글, 어떤 관점, 어떤 톤"만 결정. 그 외 시스템 결정 (어떤 도구·어떤 코드·몇 개 후보) 은 모두 CTO·시스템이 알아서.

## 🎯 CEO가 결정·조사할 것 (체계 관점)

운영 routine (매주 longlist 픽 같은 것) 은 당연히 하는 거라 제외. 시스템 체계 측면에서 진짜 결정할 것만:

| 항목 | 권고 답 | 결정 |
|---|---|---|
| **C1. 본 시스템 방향성 OK인지** | (CEO 검토 필요) | ☐ |
| **C2. 글 성과를 어떤 지표로 평가할 것인가** (Phase 3 자동 학습의 정답 데이터로 사용됨) | 페이지뷰 40% + 체류시간 30% + 재방문률 30% 가중평균 | ☐ |
| **C3. 글 발행 후 결과(트래픽 등)를 어디까지 모니터링할 것인가** | Google Search Console + GA4 (둘 다 무료) | ☐ |

→ 총 3개. 그 외는 CEO 결정 사항 X. 운영하면서 직접 픽하고 코멘트하면 시스템이 알아서 학습.

## 🛠 CTO가 결정·조사·만들 것

### 1. 결정해야 할 것

각 항목마다 권고 답 명시. CTO가 ☑ 또는 변경 의견.

| 항목 | 풀어 설명 | 권고 답 | 결정 |
|---|---|---|---|
| **운영 도구 (LLMOps)** | 프롬프트 버전 관리 + AI 호출 추적 + 비용 모니터링 통합 도구 | **Langfuse** (오픈소스, 직접 서버에 설치하면 무료) | ☐ |
| **프롬프트 관리 방식** | 5단계 각 프롬프트를 어떻게 저장·버전 관리할지 | Git에 .md 파일로 저장 + Langfuse 연동 | ☐ |
| **평가 데이터 형식** | "이 input 들어오면 이런 output 나와야 한다" 라는 정답 셋 | YAML 파일로 Git에 저장 | ☐ |
| **AI가 글 품질 평가하는 모델** | 발행 전에 AI가 자동 점검 (사실관계·thesis 명확성·후킹 강도). 6점 미만 자동 reject | Claude Sonnet 4.6 (Opus는 비용 5배인데 정확도 차이 작음) | ☐ |
| **5단계 각 단계 AI 모델 분배** | 비용 vs 품질 균형 | 2단계만 Claude Opus 4.7, 나머지는 Sonnet 4.6 | ☐ |
| **발행 플랫폼** | 네이버 블로그 / 티스토리 / 워드프레스 | 티스토리 또는 워드프레스 (구글 SEO·애드센스 단가 우위) | ☐ |
| **자동화 더 높이는 도구 (DSPy 등) 도입 시점** | AI가 프롬프트 스스로 최적화. 발행 100편 누적 후 검토 | 발행 100편 + 평가 셋 50개 모인 후 검토 (1년+) | ☐ |
| **CEO에게 보고하는 비용 표시 방식** | 글 발행 전에 "이 글 토큰 비용 약 $X" 자동 표시 | 모든 발행 prompt 끝에 비용 추정치 자동 출력 | ☐ |

### 2. 조사·점검해야 할 것

이미 한 작업이 많을 듯. 새로 정의할 필요 없이 점검만:

- **이미 연결된 API 목록 정리** — DART, KRX, 한경 컨센서스, yfinance 등. 어떤 게 연결되어 있고 작동 잘 되는지 확인
- **추가로 필요한 데이터 source 식별** — 위 점검 결과 누락된 source 있으면 보강
  - 종목 가격·재무: yfinance / FinanceDataReader (한국·미국 종목 cover)
  - 한국 공시: DART
  - 한국 거래 데이터 (외국인·기관 매매): KRX
  - 증권사 컨센: 한경 컨센서스
  - 검색량: 네이버 검색광고 + Google Trends
  - 미국 공시: SEC EDGAR
  - 뉴스: 네이버 뉴스 + (영문) Reuters/Bloomberg RSS
- **발행 플랫폼 API 또는 자동 발행 가능성 검증** — 티스토리 vs 워드프레스 자동 발행 가능 여부

→ 2~3일 내 끝나는 점검 작업.

### 3. 만들어야 할 것 (순서)

기간은 CTO가 알아서 정함. 작업 순서만 명시:

#### A. 인프라 셋업
- Langfuse 서버 설치 (오픈소스, Docker로 간단)
- 코드 저장소 초기 구조 (프롬프트 폴더·평가 셋 폴더·실행 스크립트)
- AI API 키 환경 설정

#### B. 5단계 중 가장 핵심 단계 우선 검증 ⭐
**2단계 (분석 관점 잡기)** — 시장 통념을 반박하는 thesis를 자동 생성. 이 단계가 작동 안 하면 전체 프로젝트 의미 없음. 다른 단계 만들기 전 이것부터 검증.
- 2단계용 프롬프트 초안 작성
- 평가 셋 5개 작성
- 실제 5개 종목으로 시험 → CEO에게 결과 시연
- **CEO가 "thesis 품질 OK"라고 판단해야 다음 단계 진입**. 안 되면 프롬프트 개선하거나 전체 plan 재검토

#### C. 나머지 4단계 구축
- 1단계 (글감 잡기), 3단계 (근거 수집), 4단계 (글 작성), 5단계 (다듬기)
- 각 단계 프롬프트 + 평가 셋
- 5단계 chain 작동 검증

#### D. CEO 인터페이스
- Streamlit으로 간단한 대시보드
- 후보 카드 형태 (3개 비교, 1-click 픽, 코멘트 입력)
- 각 단계 진행 상황 표시

#### E. 발행 자동화
- 매주 월요일 글감 후보 자동 산출
- CEO 픽 → 5단계 자동 진행 → 발행
- 운영 데이터 자동 누적 (CEO가 어떤 후보 픽했는지·거부했는지·코멘트 등)

#### F. 모니터링
- 발행 후 트래픽·체류시간 자동 수집 (Search Console + GA4)
- 글 1편당 토큰 비용 자동 표시
- 운영 데이터 dashboard

→ 상세 사양은 본문 Part 1·2·3 참고.

### 4. 단계별 자동화 수준 변화 (참고)

처음부터 모든 게 자동 학습되는 건 아님. 단계적 진화:

| 단계 | 무엇이 자동화되나 |
|---|---|
| **초기** | 5단계 chain 작동, CEO가 픽·검수, 데이터 누적 |
| **개선** | AI가 글 품질 자동 평가 (6점 미만 자동 reject), A/B 테스트로 어떤 프롬프트가 좋은지 비교 |
| **자동 학습** | CEO 픽 데이터로 글감 추천 알고리즘 자동 개선, 프롬프트 자동 update |
| **고도 자동화** | 프롬프트 자체를 AI가 최적화 (DSPy 같은 도구). 1년+ 후 검토 |

## Part 1. 운영 모드 — CEO 결정 vs 시스템 결정 (상세)

### 1-1. 결정 권한 분리

**CEO는 콘텐츠 의사결정만. 시스템 의사결정은 CTO·시스템 자체.**

| 결정 | 누가 | CEO 개입 |
|---|---|---|
| 어떤 글 쓸까 | CEO | ✅ |
| 어떤 thesis로 갈까 | CEO | ✅ |
| 어떤 톤·후킹 강도로 발행할까 | CEO | ✅ |
| 후보 몇 개 보여줄까 | 시스템 | ❌ |
| 어떤 프롬프트 쓸까 | 시스템 | ❌ |
| 어떤 데이터 source 쓸까 | CTO | ❌ |
| Dashboard 어떻게 만들까 | CTO | ❌ |
| 점수 가중치 얼마 | 시스템 (자동 학습) | ❌ |

### 1-2. CEO가 1편당 마주칠 결정 3가지

**결정 1. 글감 픽**

```
[A] 종목 분석 — 오라클 부채 우려 반박
[B] 산업 구조 — 엔터프라이즈 SW 가치사슬 재편
[C] 통념 비판 — RPO 지표의 함정

CEO: A 픽
코멘트 (선택): "OCI에 더 집중해줘"
```

**결정 2. 분석 관점 (thesis) 픽**

```
[1] 부채 본질이 아니라 OCI 회계 인식이 핵심
[2] capex 회수율이 진짜 변수
[3] peer group 정의 자체가 잘못됨

CEO: 1 픽
코멘트: "deferred revenue도 분석"
```

**결정 3. 제목·후킹 강도 픽 (발행 직전)**

```
[A] 강함:  "오라클 부채 우려는 거짓말이다"
[B] 중간:  "오라클 부채 우려, 진짜 문제는 다른 곳"
[C] 약함:  "오라클 부채 우려에 대한 다른 시각"

CEO: B 픽
```

→ 1편당 결정 3회. 각 30초~2분. **총 5~10분/편**.

### 1-3. 시스템이 알아서 정하는 것 (CEO에게 묻지 않음)

| 항목 | 권고치 | 근거 |
|---|---|---|
| 글감 후보 개수 | 3개 | 결정 피로 vs 다양성 균형 |
| Thesis 후보 개수 | 3개 | 동일 |
| 후킹 강도 옵션 | 3개 (强·中·弱) | 톤 spectrum |
| CEO 코멘트 처리 | 다음 단계 프롬프트에 자동 inject | 표준 패턴 |
| 거부·재생성 횟수 한도 | 3회 | 무한 루프 방지 |
| 대시보드 도구 | Streamlit | 빠른 구축 |

### 1-4. CEO 코멘트 처리

자유 코멘트 (예: "OCI에 더 집중") 가 다음 단계 프롬프트에 자동 inject:

```python
next_stage_prompt = f"""
{base_prompt}

[운영자 추가 지시]
{operator_comment}

위 지시를 반영하여 작업하라.
"""
```

코멘트가 모호해도 (예: "더 깊이") 시스템이 다시 묻지 않음. 자체 해석. 결과가 의도와 다르면 CEO가 reject + 재생성.

→ 결정 횟수 늘리는 친절은 운영 효율의 적.

## Part 2. 종목·섹터 발굴 — 단순 후보 리스트 메커니즘 (상세)

### 2-1. 방향

복잡한 점수 가중치 X. 매주 후보 15개 자동 추출 → CEO 픽 → 데이터 누적 → 자동 고도화.

### 2-2. 매주 후보 리스트 생성 로직

월요일 자동 산출. 4단계:

1. **유니버스**: KOSPI 200 + KOSDAQ 150 + S&P 500 (총 850종목)
2. **필터**: 5개 중 1개 이상 통과한 종목만 추출
   - 최근 7일 내 증권사 추정치 변경
   - 최근 7일 내 가격 ±10% 이상 변동
   - 최근 분기 EPS surprise ±20%
   - 최근 7일 내 뉴스 빈도 평균 ±2σ
   - 최근 분기 RPO·deferred revenue·재고 등 재무 이상치 (5년 평균 ±2σ)
3. **글 가치 필터**: 네이버 검색량 + Google Trends 합 상위 80%
4. **출력**: 상위 15개 (종목명 + trigger 종류 + 1줄 요약)

### 2-3. 출력 형식

```
주간 발굴 — 2026/05/05 ~ 05/11

01. 오라클 (ORCL) — 컨센 변경 + 가격 -8%
    "최근 5개사 중 3사 TP 하향. 부채 우려 헤드라인 多."

02. 에코프로 (KQ:086520) — EPS surprise -35%
    "1Q 어닝쇼크. 2차전지 sentiment 부정 80%."

03. SK하이닉스 (KS:000660) — RPO 잔고 +2.3σ
    "HBM 수주 잔고 history 최고. 가격은 박스권."

...

15. 삼성바이오로직스 — 뉴스 빈도 +2.1σ
    "FDA 이슈로 1주 뉴스 빈도 급증."
```

→ 1줄 요약. CEO 1분 내 훑어볼 분량.

### 2-4. CEO 픽 + 코멘트의 2가지 용도

```
CEO 픽: 01 오라클
코멘트: "부채는 표면 이슈, OCI 회계 인식 시점으로 가고 싶음"
```

이 코멘트가:
1. 1단계 (글감 잡기) 프롬프트에 inject — 해당 글 작성용
2. CEO 선호 패턴 데이터로 누적 — 향후 자동 고도화 input

### 2-5. 자동 고도화 (Phase 2~3)

3개월 + 100건+ CEO 픽 데이터 누적 후, 어떤 trigger가 CEO 선호 신호인지 단순 통계 모델로 자동 학습:

```python
def update_signal_weights(picks, rejections):
    X_picks = extract_trigger_types(picks)
    X_rejections = extract_trigger_types(rejections)

    weights = LogisticRegression().fit(
        np.vstack([X_picks, X_rejections]),
        np.concatenate([np.ones(len(picks)), np.zeros(len(rejections))])
    ).coef_[0]

    return weights
```

→ 사람이 미리 가중치 정하지 않음. 운영 데이터에서 역산.

### 2-6. 산업·섹터 발굴 — 별도 리스트

T1 (산업 구조 분석) 글용. 매주 1회:

- 산업별 가격 1개월 ±15% 이상 변동
- 산업별 capex 분기 ±2σ
- 정책·규제 발표 산업
- 종목 리스트에 같은 산업 3종목+ 등장

→ 상위 5개 산업 출력.

### 2-7. 헛다리 종목 거르기

- CEO 거부 → 다음주 재등장 시 우선순위 하향 (감쇠 0.5)
- 4주 연속 거부 → 일시 제외 (3개월 휴면)
- 거부 사유 코멘트 누적 → 향후 가중치 학습 input

### 2-8. CEO 직접 input

CEO가 직접 종목 던지면 (예: "이번 주는 SK하이닉스로 가자"):

- 우선 발행
- 던진 종목이 자동 리스트에 없으면 → 자동 리스트 알고리즘 검토 → 학습 데이터
