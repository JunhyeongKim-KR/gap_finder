## CEO 질문에 대한 CTO 답변

### Q1. Agent 프롬프트와 DB 분리 구조가 맞는지?

**답변: 맞다. 완전 동의.**

- Agent 프롬프트 = 정적 시스템 프롬프트 (해석 엔진)
- DB = 동적 사실 데이터 (재료 저장소)
- 이 분리는 소프트웨어 아키텍처의 기본 원칙(관심사 분리)에 정확히 부합
- 구현: `docs/` 폴더에 Agent 프롬프트 문서를 작성하고, `CLAUDE.md`에서 참조하는 구조로 설계

### Q2. 크롤링 2개 파이프라인 분리 가능한지?

**답변: 가능하고 분리해야 한다.**

- A. 글 생성용 배경지식 크롤링 → `scripts/collect_*.py` 계열 (이미 3종 존재)
- B. 모델 개선용 스타일 크롤링 → `scripts/learn_*.py` 계열 (신규)
- 목적/주기/데이터 형태가 다르므로 스키마도 테이블도 별도 설계

### Q3. 블로그 스타일+조회수 학습 구조 가능한지?

**답변: 부분적으로 가능. 단, 제약 있음.**

- 가능한 것: 블로그 URL → HTML 파싱 → 제목/구조/길이/도입방식 추출
- 제약: 조회수는 플랫폼별로 다름
  - 네이버 블로그: API 비공개, 크롤링 가능하나 불안정
  - 티스토리: 관리자 API 있지만 본인 블로그만
  - 외부 블로그 조회수: 대부분 비공개
- **1단계 결론:** 우리 블로그 성과 데이터만 먼저 추적 (자체 데이터)
- **2단계:** 벤치마크 블로그 구조 분석은 가능 (조회수 없이 스타일만)

### Q4. 데이터 항목들이 실제 스키마 설계 대상인지?

**답변: 맞다. 하지만 단계적 접근 필요.**

CEO가 정리한 A~J 범주는 최종 목표 스키마. 현실적으로:

| 우선순위 | 범주 | 이유 |
|----------|------|------|
| **즉시** | E(기업 재무), G(컨센서스) | 글 생성에 직접 필요, yfinance 등으로 자동화 가능 |
| **1개월 내** | A(거시), F(경영진), H(이벤트) | 뉴스/공시 크롤링 연동 시 |
| **2~3개월** | B(산업), D(자원) | 수동 입력 + 반자동 수집 혼합 |
| **장기** | C(정책/거버넌스), I(인프라 기업), J(스타일) | 정성 데이터 많아 구조화 난이도 높음 |
