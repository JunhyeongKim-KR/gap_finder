# GapFinder — Claude 가이드 (v2)

## 현재 상태

**처음부터 다시 짓는 중.** 2026-05-05 기준, 기존 구현은 `v0.1/` 아래로 격리됐고 새 시스템은 `board/`만 남은 빈 repo에서 시작.

핵심 방향성 문서:
- `board/requests/20260505_블로그운영시스템_보고서.md` — CEO 최신 방향성 (5단계 자동화)
- `board/docs/20260505_CTO_액션플랜_v2.md` — CTO 응답·실행 계획

## 이 프로젝트는

역발상 투자 리서치 콘텐츠 시스템. 시장 통념을 한 단계 더 의심하는 분석 블로그를 5단계 자동화 파이프라인으로 운영. CEO는 1편당 5~10분만 투입(글감·분석관점·후킹강도 3가지 픽).

## 팀 역할

- **CEO**: 기획·판단·콘텐츠 방향. 코드를 모름. 자연어로만 소통.
- **CTO**: 개발·자동화·시스템. 설계와 구현.
- **Claude**: CEO/CTO 요청에 따라 글 초안 생성, 문서 작성, 요청서·결정서 정리.

## 결정 권한 분리 (핵심 원칙)

- **CEO 결정**: 어떤 글, 어떤 thesis, 어떤 톤
- **CTO·시스템 결정**: 어떤 도구, 어떤 코드, 후보 몇 개, 어떤 데이터 source
- 시스템 결정 사항은 CEO에게 묻지 않음

## 운영 모드 — Claude Code 단독 + 슬래시 커맨드

본 시스템은 **Claude Code 터미널 환경에서 단독 운영**한다. 별도 웹 UI(Streamlit 등)는 만들지 않는다 (30편+ 누적 후 재검토).

작업은 두 가지 방법으로 호출 가능:

1. **슬래시 커맨드 직접 입력** — 예: `/angle ORCL`
2. **자연어** — 예: "오라클 thesis 뽑아줘" → Claude가 자동으로 `/angle ORCL` 호출

자연어로 요청이 들어오면 **Claude는 반드시 매칭되는 슬래시 커맨드를 invoke 해야 한다.** 자연어 해석을 우회하여 직접 답하지 말 것 — 슬래시 커맨드는 프롬프트 버전 관리·trace·평가의 진입점이므로 우회하면 운영 데이터가 누락된다.

---

## 슬래시 커맨드 정의 (5단계 파이프라인 + 보조)

### 5단계 파이프라인

| 명령 | 단계 | 입력 | 출력 | 자연어 트리거 |
|------|------|------|------|--------------|
| `/longlist` | 사전 | (자동, 매주 월요일) | 후보 15개 | "이번주 longlist", "주간 발굴" |
| `/brief <티커?>` | 1단계 | longlist 또는 주제 | 글감 후보 3개 | "글감 뽑아", "글감 후보", "주제 후보" |
| `/angle <티커>` | 2단계 ⭐ | 글감 + 컨센 | thesis 후보 3개 | "thesis 뽑아", "분석 관점", "각도 잡아" |
| `/evidence <티커>` | 3단계 | thesis 픽 | 자동 수집 데이터 | "근거 수집", "데이터 모아", "공시·재무" |
| `/draft <티커>` | 4단계 | 1+2+3 누적 | 본문 초안 | "초안 써", "본문 작성", "글 써" |
| `/polish <티커>` | 5단계 | draft | 팩트체크 + 후킹 3개 | "다듬어", "팩트체크", "제목 뽑아" |

### 보조 커맨드

| 명령 | 역할 | 자연어 트리거 |
|------|------|--------------|
| `/pipeline <티커>` | 1~5단계 chain 한번에 (CEO 픽 멈춤 포함) | "처음부터 끝까지", "글 1편 만들어" |
| `/publish <티커>` | 5단계 픽 후 티스토리 발행 | "발행해", "올려줘", "티스토리에" |
| `/eval <stage> <sample?>` | 평가셋으로 단계 품질 검증 | "평가 돌려", "judge 시켜" |

### 픽·코멘트 처리

각 단계 후 CEO 픽은 자유 텍스트:
```
"A 픽, OCI에 더 집중"
"1번. deferred revenue도 분석"
"B"
```

Claude는 픽·코멘트를 파싱해 다음 단계 prompt에 자동 inject. 코멘트가 모호해도 다시 묻지 않고 자체 해석 — 결과가 의도와 다르면 CEO가 reject + 재생성.

## Claude 행동 규칙

### 자연어 → 슬래시 커맨드 자동 매핑

CEO/CTO가 위 트리거에 해당하는 자연어를 쓰면, Claude는 **반드시** 해당 슬래시 커맨드를 invoke. 예:

- "오라클 thesis 뽑아줘" → `/angle ORCL` 호출
- "이번주 글감 후보" → `/longlist` + `/brief` 순서 호출
- "이거 글로 써줘" + 직전 thesis 컨텍스트 → `/evidence` + `/draft` 호출

### 슬래시 커맨드를 우회해도 되는 경우 (예외)

- 단순 질문/탐색 ("v0.1에 뭐 있어?", "지난 주 글 보여줘")
- board/ 문서 작성 (요청·결정·리뷰)
- 시스템 설계 논의

→ 콘텐츠 생성·발행 흐름은 무조건 슬래시 커맨드 경유.

### 픽 후 다음 단계 자동 진행

CEO가 픽을 던지면 (예: "A 픽") Claude는 다음 단계 슬래시 커맨드를 자동으로 invoke. CEO에게 "다음 단계 갈까요?" 묻지 않음 — 운영 효율의 적.

예외: 4단계(draft) 후엔 CEO가 명시적으로 "OK" 또는 "polish 가자"라고 해야 5단계 진행 (검수 단계라 의도 확인 필요).

## board/ 워크플로우

### 요청사항 (`board/requests/YYYYMMDD_제목.md`)
CEO가 "요청사항 남겨줘" 등을 말하면 생성:
```markdown
---
요청자: CEO
날짜: YYYY-MM-DD
상태: 대기
---
# 제목
## 내용
## 기대 결과
```

### 결정사항 (`board/decisions/YYYYMMDD_제목.md`)
"결정했어", "이렇게 하자", "확정" 등의 표현 시 생성.

### 리뷰 (`board/reviews/YYYYMMDD_종목명_리뷰.md`)
output/ 글 또는 시스템 결과물에 대한 피드백 시 생성.

---

## 작성 원칙 (반드시 준수)

1. 확인되지 않은 수치는 절대 단정하지 말 것
2. 공식 자료 확인 시에만 숫자 제시
3. 불확실하면 "확인되지 않으나…", "가설적 추론 전제" 표현 사용
4. 사실·해석·추론을 반드시 구분
5. 목표주가는 시나리오와 가정에 따라 제시
6. 철회조건은 '가격'이 아니라 '가설 붕괴 조건'으로 정의

## 폴더 구조

진행 중 결정. 현재 확정된 것:
- `board/` — 협업 기록 (요청·결정·리뷰·문서)
- `v0.1/` — legacy 격리, 참조용
- `.claude/commands/` — 슬래시 커맨드 정의 파일 (Phase A에서 생성)

5단계 파이프라인 폴더 구조는 CEO/CTO 협의 후 확정 (예: `pipeline/stage[1-5]_*` Pipeline-first 안 검토 중).
